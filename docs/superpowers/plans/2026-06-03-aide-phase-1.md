> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the AIDE framework skeleton — conventions, schema, gate engine, config templates, aide-spec skill, and aide orchestrator skill — so a business project can run `/aide "feature description"` to go from requirements → spec with a confirm gate and git commit.

**Architecture:** 8 files across 4 directories. `aide-core/` holds shared infrastructure (conventions, schema, gate logic). `skills/` holds Claude Code skills (orchestrator + spec stage). `templates/` holds business-project config templates. Phase 1 only implements the `confirm` gate type and tracks stage completion via file existence.

**Tech Stack:** Claude Code skill format (markdown + YAML frontmatter), JSON Schema 2020-12, YAML config

# AIDE Conventions

## Artifact Root

All AIDE workflow artifacts live under `.aide/` in the business project root. This directory should be added to `.gitignore` in the business project (AIDE's auto-commit explicitly stages `.aide/` files, so `.gitignore` does not block them).

## Output Structure

```
.aide/
├── config.yaml          # Project workflow configuration (copied from template)
├── state.json           # Pipeline state tracking (Phase 2+)
└── output/
    ├── 1-spec/
    │   ├── spec.md      # Human-readable specification
    │   └── spec.json    # Machine-readable specification (conforms to spec.schema.json)
    ├── 2-plan/
    │   ├── plan.md
    │   └── plan.json
    ├── 3-implement/
    │   └── implement.json
    └── 4-test/
        ├── test-report.md
        └── test-report.json
```

## Stage Order

| Order | Stage     | Description                  |
|-------|-----------|------------------------------|
| 1     | spec      | Requirements → Specification |
| 2     | plan      | Specification → Plan         |
| 3     | implement | Plan → Code changes          |
| 4     | test      | Verification → Test report   |

## File Naming

- Stage directories: `{order}-{stage-name}/` (e.g., `1-spec/`)
- Human-readable artifact: `{stage-name}.md`
- Machine-readable artifact: `{stage-name}.json`

Every stage MUST produce both `.md` and `.json` outputs. The `.json` output must conform to the corresponding schema in `aide-core/schemas/`.

## Git

After each stage completes, AIDE auto-commits only `.aide/` files with message format:

```
aide(<stage>): <summary>
```

Business code changes are never auto-committed. Working-tree changes outside `.aide/` produce a warning but do not block the commit.
```

# Gate Engine

The gate engine controls pipeline checkpoints — points where AIDE pauses for human confirmation before proceeding to the next stage.

## How It Works

1. After a stage completes, the orchestrator reads the gate configuration for that stage from `.aide/config.yaml`.
2. For each configured gate, it presents the stage's artifact summary and the gate's prompt.
3. The user responds, and the engine resolves the gate.

## Gate Types

### confirm

The pipeline halts. The user must explicitly approve by typing `y` or `yes`. If the user types `n` or `no`, the pipeline stays at the current stage. The user can provide feedback (e.g., "no, the acceptance criteria for F002 are too vague"), and the orchestrator will re-invoke the stage skill with that feedback.

### confirm_skip (Phase 2)

The pipeline pauses but the user can type `skip` or `s` to bypass without approving. Typing `y`/`yes` approves; `n`/`no` rejects with feedback.

### auto (Phase 3)

No pause. The gate is logged and the pipeline proceeds immediately. Useful for CI or high-trust stages.

## Gate Resolution Algorithm

```
for each gate in stage_config.gates:
    if gate.type == "confirm":
        display gate.prompt
        wait for input
        if input is "y" or "yes":
            continue to next gate (or next stage if last gate)
        else if input is "n" or "no":
            ask for feedback
            re-invoke current stage with feedback
            restart gates for this stage
    elif gate.type == "confirm_skip":
        display gate.prompt
        wait for input
        if input is "y" or "yes": continue
        else if input is "skip" or "s": continue
        else if input is "n" or "no": ask feedback, re-invoke stage
    elif gate.type == "auto":
        log "Gate: {gate.name} (auto) - passed"
        continue
```

## Phase 1 Scope

Only the `confirm` gate type is implemented. The `confirm_skip` and `auto` types are documented here for forward reference but the orchestrator treats unknown types as `confirm` with a warning.
```

# aide-spec — Requirements to Specification

You are the spec-writing stage of the AIDE development pipeline. Your job: take a raw requirement description and turn it into a structured, verifiable specification.

## Input

The user will provide a requirement description (either directly or passed through from the `/aide` orchestrator). This may be a short sentence or a detailed paragraph.

## Process

1. Analyze the requirement. Identify distinct features, edge cases, and non-obvious implications.
2. Clarify ambiguity proactively — if the requirement is vague on something important, ask the user before writing the spec. One question at a time.
3. Draft the spec following the output conventions below.

## Output

Write two files:

### `.aide/output/1-spec/spec.md`

A human-readable specification document covering:

- **Overview** — one paragraph summarizing the feature
- **Features** — each feature with its acceptance criteria
- **Constraints** — technical or business constraints
- **Scope boundary** — what is explicitly NOT included

### `.aide/output/1-spec/spec.json`

A machine-readable specification conforming to the JSON Schema at `aide-core/schemas/spec.schema.json`. The schema requires:

```json
{
  "features": [
    {
      "id": "F001",
      "title": "...",
      "description": "...",
      "acceptance_criteria": ["..."]
    }
  ],
  "constraints": ["..."],
  "scope_boundary": "..."
}
```

Feature IDs follow the format `F001`, `F002`, etc. Every feature must have at least one acceptance criterion.

## Conventions

Read `aide-core/conventions.md` for the artifact directory layout and naming rules. Follow them exactly.

## Schema Validation

After writing `spec.json`, validate it against `aide-core/schemas/spec.schema.json`. If validation fails, fix the output and re-validate before reporting completion.

## Completion

When both files are written and validated, report a summary:

```
[aide-spec] Specification complete:
  - Features: N
  - Constraints: N
  - Scope boundary: <summary>
  - Output: .aide/output/1-spec/
```
```

# aide — Development Pipeline Orchestrator

You are the orchestrator of the AIDE development pipeline. You do NOT write specs, plans, or code yourself. Your job is to: load configuration, invoke stage skills in order, run gate checkpoints, and handle git commits.

## Pipeline Stages

1. **spec** — skill: `aide-spec` — Requirements → Specification
2. **plan** — skill: `aide-plan` (Phase 2) — Spec → Plan
3. **implement** — skill: `aide-implement` (Phase 2) — Plan → Code
4. **test** — skill: `aide-test` (Phase 3) — Verification

## Startup

When the user invokes `/aide <requirement description>`:

1. Read conventions from `aide-core/conventions.md` (located in the AIDE installation directory, typically `.claude/aide/aide-core/conventions.md`).
2. Load configuration:
   - If `.aide/config.yaml` exists in the business project root, parse it.
   - If not, use defaults: all enabled stages use `confirm` gates.
3. Set the starting stage to `spec` (stage 1).

## Stage Execution

For each stage from the starting stage through test:

1. **Check if stage is enabled** in config. If `enabled: false`, skip to next stage.
2. **Display progress**: `[aide] Stage 1/4: spec — Generating specification...`
3. **Invoke the stage skill**: Call the corresponding skill by name (e.g., `aide-spec`) with the requirement or previous stage's output as context.
   - For spec stage: pass the user's original requirement description.
   - For later stages: point the skill to the previous stage's output files in `.aide/output/`.
4. **Verify output**: Check that the stage's output files exist in the expected location (e.g., `.aide/output/1-spec/spec.md` and `.aide/output/1-spec/spec.json`).

## Gate Checkpoint

After a stage produces output:

1. Read the gate configuration for that stage from config.
2. For each gate:
   - Read the stage's `.md` output and present a structured summary (not the full text).
   - Present the gate's `prompt`.
   - **confirm** type: Wait for `y`/`yes` (proceed) or `n`/`no` (stay).
   - If user rejects (`n`/`no`): ask for brief feedback, then re-invoke the same stage skill with that feedback. Loop back through gates when it completes.
3. All gates passed → proceed to git commit.

## Git Commit

After gates pass:

1. Run `git status` to check the working tree.
2. If files outside `.aide/` are modified but uncommitted, warn the user: `[aide] Warning: uncommitted changes exist outside .aide/. These will NOT be included in this commit.`
3. Stage and commit only `.aide/` files:
   ```bash
   git add .aide/output/<current-stage>/ .aide/state.json 2>/dev/null
   git commit -m "aide(<stage>): <summary>"
   ```
   - Summary format for spec: `aide(spec): <concise feature summary from spec>`
4. Report: `[aide] Committed: aide(spec): <summary>`

## Completion

After the final stage (and any final gate) completes:

```
[aide] Pipeline complete:
  ✓ spec  — F001, F002 (commit: abc1234)
  ✓ plan  — 4 tasks (commit: def5678)
  ✓ implement — 3 files changed (commit: ghi9012)
  ✓ test  — 4/4 pass (commit: jkl3456)
```

## Error Recovery

- If a stage fails (no output files, or output fails schema check): report which stage failed, show the error, and ask: "Retry / Skip / Abort?"
- If user interrupts: tell them they can resume with `/aide --continue`.

## Config Defaults

When `.aide/config.yaml` is missing, assume:

```yaml
version: "1"
strict_mode: false
stages:
  spec:
    enabled: true
    gates:
      - after_spec:
          type: confirm
          prompt: "Review the spec. Does this look right? (y/n)"
```

## Phase 1 Scope

Only stage 1 (spec) is active. All other stages are `enabled: false` in the template config. Do not attempt to invoke them.
```

# AIDE — AI-Driven Development Automation

A Claude Code skill collection for structured, AI-driven development workflows. Business projects reference AIDE via git submodule to add `/aide` — a pipeline that takes a requirement and runs it through spec → plan → implement → test with human gates at each stage.

## Quick Start

### Add AIDE to your project

```bash
cd your-project/
git submodule add <AIDE-repo-url> .claude/aide
cp .claude/aide/templates/aide.config.yaml .aide/config.yaml
```

Then add this line to your project's `CLAUDE.md`:

```yaml
extra_skill_dirs: [.claude/aide/skills]
```

### Run the pipeline

```bash
/aide "Add user login with OAuth support"
```

AIDE will:
1. Generate a structured spec (`.aide/output/1-spec/`)
2. Pause for your review
3. On confirm, commit the spec and proceed to planning

### Customize gates

Edit `.aide/config.yaml` to change gate types per stage:

- `confirm` — requires explicit y/n
- `confirm_skip` — can be skipped (y/n/skip)
- `auto` — no pause
